## 1. Основные положения и структура

Код вашего бизнес-сервиса хранится в директории `src`. Директория `aigw-project` - это непосредственно директория клонирования проекта с bitbucket.

При разработке бизнес-сервиса вам следует использовать собственное название. Это касается как директории `src/aigw_service`, так и настроек в разделе `[tool.poetry]` (поля `name` и `packages`) в файле `pyproject.toml`.

Представленная структура является рекомендацией. Вы вправе изменять её по своему усмотрению. Однако если вы решили придерживаться предложенной структуры, учтите, что некоторые файлы лучше не редактировать самостоятельно (например, `src/aigw_service/api/os_router.py`). Такие файлы помечены соответствующими комментариями. Также не забывайте удалять неиспользуемый функционал (например, демонстрационный шаблонный роут, соответствующий требованиям GenAI Ready API, в `src/aigw_service/api/v1/routers.py`).

Мы всегда готовы оказать поддержку и проконсультировать по вопросам структуры и работы данного шаблона. **_Базовая работоспособность шаблона проверяется при каждом внесении изменений с нашей стороны_**. При возникновении ошибок в первую очередь проверяйте логику вашего кода и его интеграцию в структуру шаблона, проводите тщательную отладку, пишите и запускайте тесты 😊

1. Все доменные директории должны находиться в `src`.
2. `src/aigw_service` - корневой уровень приложения, содержащий конфигурации, константы и т.д.
    1. `__main__.py` - точка входа для запуска FastAPI-приложения.
    2. `config/*` - обработка переменных окружения и других настроек.
    3. `base.py` - базовые классы для работы приложения.
    4. `context.py` - реализация паттерна AppContext для управления жизненным циклом компонентов приложения. Основная идея паттерна заключается в создании объекта, содержащего ссылки на сервисы, ресурсы и настройки, доступные из любой части приложения. Этот объект создается один раз при запуске приложения и передается в методы или конструкторы других объектов. Во избежание проблем с глобальным доступом к публичным ресурсам класса следует реализовывать открытые функции, которые будут выступать зависимостями для методов вашего REST-приложения.
3. `aigw_service/logger` - реализация логгера приложения
    1. `context_vars.py` - контекстные переменные для сквозного логирования. Приходящие в запросе заголовки сохраняются здесь и используются на протяжении обработки запроса Механизм контекстных переменных позволяет асинхронно обрабатывать несколько запросов и не смешивать их основные параметры (например, Request-Id). *Редактировать не рекомендуется*.
    2. `logger.py` - реализация класса логгера. *Редактировать  не рекомендуется*.
    3. `models.py` - модели для объектов логирования, метрик и событий аудита. *Редактировать запрещено*.
    4. `utils.py`- вспомогательные функции для логгера и настройка маскирования.
    5. `uvicorn_logging_config.py`- конфигурация встроенного в uvicorn логгера.
4. `aigw_service/api` - уровень реализации бизнес-логики.
    1. `__init__.py` - конфигуратор FastAPI приложения. Здесь настраивается версионирование API и методы, включаемые в релиз.
    2. `os_router.py` - глобальные инфраструктурные методы: health check и info. *Редактировать запрещено*.
    3. `metric_router.py` - глобальные инфраструктурные методы для сбора метрик. *Редактировать запрещено*.
    4. `schemas.py`- pydantic схемы для инфраструктурных методов. *Редактировать запрещено*.
    5. `middleware.py`- логика перехвата сообщений FastAPI. Обеспечивает запись логов, событий аудита и метрик.
       *Редактировать запрещено*.
5. `aigw_service/api/v1` - версия бизнес логики.
    1. `routers.py` - ядро со всеми эндпоинтами (ручками).
    2. `schemas.py` - Pydantic-модели для описания ответов, тел запросов, возвращаемых ошибок и т.д.
    3. `models.py` - модели для работы с бд, дата классами, именованными кортежами и т.д.
    4. `services.py` - бизнес-логика, специфичная для сервиса.
    5. `utils.py` - вспомогательные функции, не связанные с бизнес-логикой (нормализация ответов, обогащение данных и т.д.).
    6. `exceptions.py` - исключения, относящиеся к сервису (например, `PostNotFound`, `InvalidUserData`).
    7. `agent_examples/*` - примеры реализации ИИ-агентов на базе платформенных инструментов.

## 2. Переменные окружения

Подробное описание переменных окружения доступно в [соответствующем разделе документации на confluence](https://confluence.sberbank.ru/pages/viewpage.action?pageId=15956356984)

Необходимо создать файл `.env` на одном уровне с директорией `src`. В него следует добавить базовые переменные окружения, а также дополнительные переменные в соответствии с потребностями проекта (актуальный набор переменных представлен в документации по ссылке выше).

Вы также можете использовать другие способы передачи переменных окружения, обеспечив их корректную загрузку приложением.

### Базовый набор переменных окружения

| Имя переменной       | Пример значения (на стенде может отличаться)        | Описание                                                                                   |
|----------------------|-----------------------------------------------------|--------------------------------------------------------------------------------------------|
| `APP_HOST`           | `0.0.0.0`                                           | Хост HTTP-сервера (локально). На Windows учитывайте `127.0.0.1`.                           |
| `APP_PORT`           | `8080`                                              | Порт HTTP-сервера сервиса.                                                                 |
| `APPLICATION_NAME`   | `aigw-giga-winner`                                  | Имя вашего бизнес-приложения.                                                              |
| `AUDIT_LOG_FILE_NAME`| `events.log`                                        | Имя файла для записи событий аудита.                                                       |
| `AUDIT_LOG_PATH`     | `/var/log/audit`                                    | Путь к директории для хранения событий аудита.                                             |
| `DEBUG`              | `True`                                              | Режим отладки (снижает уровень логирования до DEBUG).                                      |
| `HOST_IP`            | `10.3.152.161`                                      | IP-адрес Pod для системы аудита.                                                           |
| `HOST_UID`           | `63bd6cbe-170b-49bf-a65c-3ce967398ccd`              | Уникальный идентификатор Pod для системы аудита.                                           |
| `LOCAL`              | `True`                                              | Флаг, указывающий на локальную разработку.                                                 |
| `LOG_FILE_NAME`      | `app.log`                                           | Имя файла для записи логов приложения.                                                     |
| `LOG_PATH`           | `/var/log`                                          | Путь к директории для хранения логов.                                                      |
| `LOG_ROTATION`       | `10 MB`                                             | Максимальный размер файла лога перед ротацией.                                             |
| `METRIC_FILE_NAME`   | `app-metric.log`                                    | Имя файла для записи метрик.                                                               |
| `METRIC_PATH`        | `/var/log`                                          | Путь к директории для хранения метрик.                                                     |
| `PROJECT_NAME`       | `AIGATEWAY`                                         | Имя проекта в OpenShift.                                                                   |

Скопируйте базовые переменные в файл `.env` из примера ниже:

```dotenv
# Переменные приложения
DEBUG=True
LOCAL=True
APP_HOST=0.0.0.0
APP_PORT=8080

# Специфичные для OpenShift переменные
APPLICATION_NAME=<бизнес-имя приложения>
PROJECT_NAME=aigateway

# Логи
LOG_PATH=.
LOG_FILE_NAME=app.log
LOG_ROTATION=10 MB

# Метрики
METRIC_PATH=.
METRIC_FILE_NAME=app-metric.log

# Аудит
AUDIT_LOG_PATH=.
AUDIT_LOG_FILE_NAME=events.log
HOST_IP=127.0.0.1
HOST_UID=63bd6cbe-170b-49bf-a65c-3ce967398ccd
```

### ОСОБЕННОСТИ ЛОКАЛЬНОЙ РАЗРАБОТКИ С СЕРТИФИКАТАМИ

Особое внимание уделите переменной окружения `LOCAL`. Она предназначена для разграничения порядка подключения к другим АС при локальной разработке и на стендах. Если вам требуется отправлять запросы в стороннюю АС, локально потребуются соответствующие сертификаты, которые подставляются в коде. На стендах (включая ИФТ) сертификаты в коде подставлять не нужно — это выполняется автоматически на стадии **egress** (выход трафика из pod).

> _**Соединение на стендах не установится, если вы будете подставлять сертификаты в коде или будете использовать https вместо http**_.  

Для разграничения логики и предназначена переменная `LOCAL`. Вы можете реализовать это разграничение по своему усмотрению. Мы рекомендуем использовать булеву переменную `LOCAL`.

Пример применения переменной окружения `LOCAL` для переключения протокола:
```python
class BaseAppSettings(BaseSettings):
    """
    Базовый класс для настроек.
    """

    local: bool = Field(validation_alias="LOCAL", default=False)
    debug: bool = Field(validation_alias="DEBUG", default=False)

    @property
    def protocol(self) -> str:
        """Протокол HTTP."""
        return "https" if self.local else "http"
```

Пример применения переменной окружения `LOCAL` для настроек GigaChat в проекте:
```python
class GigaChatSettings(BaseAppSettings):
    """
    Настройки GigaChat.
    """

    host: str = Field(validation_alias="GIGACHAT_HOST")
    port: str = Field(validation_alias="GIGACHAT_PORT")
    endpoint: str = Field(validation_alias="GIGACHAT_ENDPOINT", default="/v1")
    tls_cert_filepath: Optional[str] = Field(validation_alias="GIGACHAT_TLS_CERT_FILEPATH", default="")
    key_filepath: Optional[str] = Field(validation_alias="GIGACHAT_KEY_FILEPATH", default="")
    ca_bundle_filepath: Optional[str] = Field(validation_alias="GIGACHAT_CA_BUNDLE_FILEPATH", default="")

    ...

    @property
    def certs(self) -> dict:
        """Сертификаты для подключения."""
        _certs = {}
        if self.local:  # self.local берется из BaseAppSettings и равен `LOCAL` из переменных окружения
            _certs = {
                "ca_bundle_file": self.ca_bundle_filepath,
                "cert_file": self.tls_cert_filepath,
                "key_file": self.key_filepath,
            }
        return _certs

    ...
```

## 3. Журналирование

Подробное описание журналирования доступно в [документации на confluence](https://confluence.sberbank.ru/pages/viewpage.action?pageId=15293458056).

В основе логирования используется [`loguru`](https://loguru.readthedocs.io/en/stable/overview.html#take-the-tour).

В шаблоне реализовано базовое логирование (журналирование): имеется объект логгера, который хранится в контексте приложения (`src/aigw_service/context.py`). Его можно импортировать в любое место приложения. Также реализовано логирование входящих и исходящих запросов через FastAPI. Подробнее с этой логикой можно ознакомиться в `src/aigw_service/api/middleware.py`, где происходит перехват запросов до их обработки основным кодом приложения и после её завершения.

Логи записываются в консоль и в файл (путь к файлу задается в переменных окружения). На стенде fluent-bit будет считывать логи из этого файла и отправлять их в АС Тенгри, где их можно просмотреть через UI.

### ЗАПРЕЩЕНО ПИСАТЬ ИНФОРМАЦИЮ УРОВНЯ К2 И ВЫШЕ В ЛОГИ

Если необходимо записывать в логи сообщения, содержащие чувствительную информацию, обратитесь к `src/aigw_service/logger/utils.py`. Там находится функция для маскирования и настройки маскирования. По умолчанию тела входящих и исходящих сообщений (в поле `args.message`) маскируются полностью. Через конфигурацию в том же файле это можно изменить, предварительно ознакомившись с разделом о маскировании в документации на Confluence.

От вас требуется реализовать логирование вашего кода. Делать это можно на ваше усмотрение, однако чем подробнее, яснее и последовательнее будут логи, тем проще будет их анализировать и находить ошибки.

Пример из `src/aigw_service/api/v1/routers.py`(логирование также рекомендуется добавлять внутри вызываемых методов):
```python
# файл src/aigw_service/api/v1/routers.py, эндпоинт example_genai_ready_api_endpoint
from aigw_service.context import APP_CTX

from .schemas import ExampleGenAIReadyAPIResponse

logger = APP_CTX.get_logger()

...
@router.post(...)
async def example_genai_ready_api_endpoint(...) -> ExampleGenAIReadyAPIResponse:
    ...
    logger.info("GenAI Ready example endpoint called")
    return ExampleGenAIReadyAPIResponse(
        response_field_1=request.request_field_1,
        response_field_2=request.request_field_2,
    )
```

Таким образом, для записи лога достаточно вызвать у объекта логгера метод, соответствующий уровню логирования, и передать сообщение. Оно будет записано в поле `message` схемы логирования. 

Уровни логирования:

| Уровень   | Значение | Метод логгера      |
|-----------|----------|--------------------|
| `TRACE`   | 5        | `logger.trace()`   |
| `DEBUG`   | 10       | `logger.debug()`   |
| `INFO`    | 20       | `logger.info()`    |
| `SUCCESS` | 25       | `logger.success()` |
| `WARNING` | 30       | `logger.warning()` |
| `ERROR`   | 40       | `logger.error()`   |
| `CRITICAL`| 50       | `logger.critical()`|

В данном шаблоне логгер автоматически заполняет основные поля, включая поле `args` при обработке входящих и исходящих сообщений.

Обратите внимание, что в консоли также отображаются логи `uvicorn` (выделены красным цветом). Они не записываются в файл и не отображаются в UI. Логи uvicorn предназначены для отладки и настраиваются через `src/aigw_service/logger/uvicorn_logging_config.py`. Настройки прописываются в `src/aigw_service/__main__.py`. При необходимости запись этих логов в консоль можно отключить.

Кроме того, с помощью логгера реализована запись событий аудита и метрик (метрики находятся в стадии разработки, но механизм уже заложен). Об **аудите** и **метриках** читайте далее.

## 4. Аудит

Подробное описание аудита доступно в [документации на confluence](https://confluence.sberbank.ru/pages/viewpage.action?pageId=15956359062).

Аудит требует записи полных тел входящих и исходящих сообщений в специальную систему. Это необходимо для безопасников и выполняется только на стендах. ### Запуск линтеров

В качестве инструментов для проверки кода используются линтеры [`ruff`](https://docs.astral.sh/ruff/) или [`pylint`](https://pylint.readthedocs.io/en/stable/). Конфигурации линтеров описаны в `pyproject.toml` в секциях `[tool.ruff]` и `[tool.pylint.*]`. Все команды для `src` выполняются из корневой директории клонирования (по умолчанию `rest_template`).

Запуск `pylint`:
```bash
pylint src
```

`pylint` должен набирать более 7 баллов! Обратите внимание, что "из коробки" `pylint` дает больше 9 баллов.

Запуск `ruff`:
```bash
ruff check src
```

Также можно запустить `ruff` и автоматически исправить возникшие несоответствия в коде:
```bash
ruff check --fix src
```

С помощью `ruff` так же можно отформатировать код:
```bash
ruff format src
```

Вместо `ruff` можно использовать [`isort`](https://pycqa.github.io/isort/) и [`black`](https://pypi.org/project/black/). `isort` организует импорты в правильном порядке, а `black` выполняет автоматическое форматирование кода.

```bash
isort src
black src
```

> `black` по умолчанию форматирует списки, параметры и аргументы в одну строку. Для многострочного форматирования добавьте завершающую запятую (trailing comma) в конце объекта.  

Конфигурации `black` и `isort` описаны в `pyproject.toml` в секциях `[tool.black]` и `[tool.isort]`.
